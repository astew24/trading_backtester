Design

Built around what makes backtests fail

Most backtesting frameworks let you add strategies. This one starts from the assumption that transaction costs, lookahead bias, and overfitting are the primary reasons backtests don't transfer. Every design decision reflects that.

Liquidity gating

Mean-reversion signals are filtered by average dollar volume — avoids false fills in illiquid names where spread cost exceeds signal edge.

Next-bar execution

Signals generated on bar close are executed on the next bar's open. Eliminates look-ahead bias from same-bar fills on daily data.

Conservative exit logic

Stop-loss, take-profit, and trailing stop levels are applied with conservative daily OHLC assumptions when multiple levels trigger on the same bar.

Benchmark-relative reporting

Every run computes excess return, tracking error, information ratio, beta, and alpha against a configured benchmark series.

Reproducible artifacts

Each backtest writes a timestamped artifact bundle: metrics.json, trades.csv, config_snapshot.yaml, and charts.

Vectorized signal generation

Strategy logic runs on full price DataFrames with pandas/NumPy vectorization — no row-by-row loops in signal paths.

Transaction Cost Model

Three-component friction model

Each trade applies commission, spread, and slippage independently. The defaults are calibrated to realistic retail/institutional costs for liquid U.S. equities on daily data.

Parameter Default Description
commission_bps 5.0 Per-trade commission as basis points of notional value
fixed_commission $1.00 Flat per-order fee, combined with bps cost
spread_bps 2.0 Half-spread paid as market impact on entry and exit
slippage_bps 1.0 Additional fill slippage beyond quoted spread
stop_loss 8% Max adverse excursion before hard exit
trailing_stop 10% Drawdown from peak triggering exit on profitable positions

Walk-Forward Validation

Parameter selection without lookahead

Walk-forward mode rolls a training window across the full data history. In each fold, the best parameter set is selected on in-sample performance, then evaluated on the immediately following out-of-sample window. Reported metrics come only from the out-of-sample periods.

This guards against the most common failure mode: curve-fitting to historical noise that doesn't repeat out-of-sample.

  • Training and test windows configured independently in config.yaml
  • Parameter grid defined per-strategy — searched exhaustively on training window
  • Selection criterion is configurable (Sharpe, Calmar, total return)
  • Each fold writes a row to walk_forward_windows.csv for audit
  • Aggregate OOS curve reflects realistic parameter drift over time

Performance Metrics

Full risk-adjusted reporting

Every run computes the full set of metrics used in quantitative strategy review:

  • Total return and annualized return
  • Annualized volatility
  • Sharpe ratio (excess return / vol)
  • Sortino ratio (excess return / downside vol)
  • Calmar ratio (annualized return / max drawdown)
  • Maximum drawdown and drawdown duration
  • Beta and alpha vs. benchmark
  • Information ratio and tracking error
  • Win rate, profit factor, average trade return
  • Average win and average loss per trade
  • Historical Value at Risk (95% confidence)
  • Average daily turnover and gross exposure

Code

Mean-reversion strategy and walk-forward run

The mean-reversion strategy scores each security by how far its price has deviated from a rolling mean, normalized by recent volatility. Signals are liquidity-gated by average dollar volume before any position is taken.

src/trading_backtester/strategies.py python 
@dataclass
class MeanReversionStrategy(BaseStrategy):
    lookback: int = 20
    entry_z: float = 1.5
    exit_z: float = 0.5
    adv_filter_days: int = 20
    min_adv: float = 1_000_000.0   # $1M minimum avg dollar volume
    allow_short: bool = True

    def generate_signals(self, data: pd.DataFrame) -> pd.DataFrame:
        frame = _validate_input_data(data)
        close = frame["Close"]
        volume = frame["Volume"]

        # Liquidity gate — skip signals on thin trading days
        adv = (close * volume).rolling(
            self.adv_filter_days, min_periods=self.adv_filter_days
        ).mean()
        liquid = adv >= self.min_adv

        # Z-score: deviation from rolling mean, normalized by rolling std
        rolling_mean = close.rolling(self.lookback).mean()
        rolling_std  = close.rolling(self.lookback).std(ddof=1)
        z_score = (close - rolling_mean) / rolling_std.replace(0, np.nan)

        target_position = pd.Series(0.0, index=frame.index)
        # Enter long when price is depressed and liquidity is adequate
        target_position = target_position.mask(
            (z_score < -self.entry_z) & liquid, 1.0
        )
        if self.allow_short:
            target_position = target_position.mask(
                (z_score > self.entry_z) & liquid, -1.0
            )

Limitations

What this does not model

This is a daily-bar simulator. It is more realistic than most toy scripts, but not a substitute for an execution simulator built on tick or order-book data.

  • No borrow availability or hard-to-borrow fee model for short positions
  • No portfolio-level cash sharing — multi-symbol runs are equal-weighted aggregates
  • No intraday event queue or partial-fill model
  • Yahoo Finance data quality varies across symbols and corporate action handling
